# 文档内容质量审查与增强

对 `docs/` 目录下的**手写笔记**进行质量审查，并自动增强代码注释与示例。

## 核心原则

这些笔记由用户亲手整理，TOC 标题结构反映的是用户的个人思路。因此：

- **格式排版**：自动修复，无需确认
- **代码增强**（补注释、补示例）：自动执行，完成后汇总改动供 review
- **TOC / 标题结构**：只标记问题，**必须先征得用户确认才能修改**
- **技术事实**：查证后直接修正错误

## 严重程度定义

| 级别 | 含义 | 行动 |
|------|------|------|
| 🔴 必须修复 | 技术事实错误、代码引用失效、会误导读者 | 审查后立即修复 |
| 🟡 建议修复 | 格式偏差、内容可优化、注释缺失 | 自动修复（格式/代码增强）或标记待确认（TOC） |
| 🟠 待确认 | TOC/标题结构调整建议 | 标记问题，用 `AskUserQuestion` 征得用户同意后修改 |
| ✅ 通过 | 无问题 | 无需操作 |

## 审查流程

### 第一步：确认审查范围

使用 `AskUserQuestion` 确认以下信息：

1. **审查哪些文件**：具体文件路径，或「最近修改的文档」
2. **审查模式**：
   - **全量审查**：对整个文件执行 7 个维度检查（默认）
   - **增量审查**：仅审查 `git diff` 中涉及变更的 H2/H3 小节。执行方式：先用 `Bash` 运行 `git diff --stat HEAD~1 -- docs/` 确认变更范围，再用 `Grep` 定位变更行所属的 H2/H3 区间，后续维度仅检查这些区间。

### 第二步：并行收集数据

以下操作尽量并行执行，不要串行等待。

同时发起以下调用：

- **`Read`** 读取目标文档全文
- **`Grep`** 提取所有 H2/H3 标题（`^#{2,3} `）
- **`Grep`** 提取所有 `--8<--` 代码引用
- **`Grep`** 提取学习目标列表（`本文你会学到` 所在行）
- **`Read`** 读取 `zensical.toml` 的 `nav` 部分（用于维度 6 校验）
- **`Bash`** 检查验证事实缓存文件是否存在（`.claude/doc-review-cache.json`）
- **explore agent**（后台）验证所有代码引用标记是否存在

### 第三步：执行审查（按以下 7 个维度检查）

维度 1 的 Context7 查询与维度 5 的代码引用验证可并行。

---

## 审查维度 1：技术准确性

**目标**：确保文档中的定义、语法、API 用法、版本号与官方文档一致。

**方法**：

1. 提取文档中涉及的核心技术断言（定义、语法说明、API 行为描述等）
2. **重点关注**：版本号声明（「Java X 引入」「JDK X.Y 新增」）、语法/命令格式、默认值/行为描述
3. 先检查本地缓存 `.claude/doc-review-cache.json` 中是否已有验证记录，命中则跳过查询
4. 未命中的断言使用 Context7 MCP 查证：
   - 先调用 `context7-resolve-library-id` 解析库 ID
   - 再调用 `context7-query-docs` 查询具体 API 的 `@Since` 标签和行为描述
   - 多个独立查询可并行发起
5. 对无法通过 Context7 查证的断言，使用 `WebSearch` 查找官方文档确认
6. 查证完成后，将新验证的事实追加到 `.claude/doc-review-cache.json`

**缓存文件格式**（`.claude/doc-review-cache.json`）：

```json
{
  "facts": [
    {
      "assertion": "FileReader(String, Charset) 是 Java 11 引入的",
      "correct": "@Since: 11",
      "source": "Context7 Java SE 21 API",
      "verified_at": "2025-01-15"
    }
  ]
}
```

**常见错误模式**：

| 错误类型 | 示例 | 查证方法 |
|---------|------|---------|
| 版本张冠李戴 | 「FileReader(Charset) 是 Java 18 引入的」（实为 Java 11） | Context7 查 `@Since` |
| 默认值错误 | 「BufferedInputStream 默认缓冲区 4KB」（实际是 8KB） | Context7 查源码常量 |
| 语法格式不规范 | 命令参数顺序或格式与 man page 不一致 | WebSearch 查官方文档 |
| 行为描述不准确 | 「PrintStream 默认自动刷新」（需构造时传 autoFlush=true） | Context7 查构造函数 |

**输出格式**：

```
🔴 [准确性] 第 X 行：「原文引用」
   事实：<正确描述>（来源：Context7 @Since / 官方文档）
   修复建议：<具体修改>
```

## 审查维度 2：内容清晰度

**目标**：确保手写笔记中的定义、术语、概念描述清晰自洽，读者能理解。

手写笔记不需要强制教学模板（类比优先、Problem→What→How），但需要确保内容本身是清晰的。

**检查项**：

| 检查项 | 说明 | 审查方法 |
|--------|------|---------|
| 术语有解释 | 专有名词首次出现时是否有括号解释或上下文说明 | 搜索技术术语首次出现位置 |
| 定义准确完整 | 记录的定义是否包含关键限定条件 | 对比官方文档 |
| 上下文连贯 | 读者能否仅从本节内容理解当前概念，不需要外部查找 | 通读每个 H3 小节判断 |
| 代码块有引导 | 代码示例前是否有简要说明（一两句即可，不需要长篇引导语） | 检查代码块前一段文字 |

**豁免**：`docs/document-translation/` 下的翻译文档忠于原书，跳过此维度。

**输出格式**：

```
🟡 [清晰度] 第 X 行 H3「标题名」：
   问题：<描述>
   建议：<改进方向>
```

---

## 审查维度 3：TOC 大纲结构（仅标记，不自动修改）

**目标**：检查标题层级是否合理，但**尊重用户的个人组织思路，不擅自修改**。

**检查项**：

1. **提取所有 H2/H3 标题**（用 `Grep` 匹配 `^#{2,3} `）
2. **检查 H4 可见性**：Zensical TOC 只显示 H2+H3，H4 在右侧目录中不可见。如果某 H3 下有 ≥3 个 H4 子节且内容独立，标记为建议提升
3. **检查明显问题**：孤立术语（H2 标题为纯类名无上下文）、万能标题（反复出现「概述」「其他」）

**行为规则**：

- 发现 TOC 问题时，**标记为 🟠 待确认**，不直接修改
- 在报告中列出建议，用 `AskUserQuestion` 逐条征求用户意见
- 用户确认后才执行标题层级调整

**输出格式**：

```
🟠 [TOC 结构] H2「标题」：
   建议：<调整方案>
   原因：<为什么建议调整>
   （等待用户确认）
```

---

## 审查维度 4：学习目标完整性

**目标**：如果文档有「本文你会学到」列表，确保它与实际 H2 章节一一对应。

**行为规则**：

- 如果文档**没有**「本文你会学到」列表，跳过此维度（手写笔记不强制要求）
- 如果**有**该列表，则执行交叉对比

**检查方法**：

1. 提取列表中的条目（匹配 `🎯 本文你会学到` 和 `本文你会学到`）
2. 提取所有 H2 标题
3. 交叉对比：遗漏 / 空头承诺 / 措辞偏差

---

## 审查维度 5：代码引用有效性

**目标**：确保所有 `--8<--` 代码片段引用指向存在的文件和标记。

**检查方法**：

1. 用 `Grep` 提取文档中所有 `--8<-- "..."` 引用路径
2. 对每个路径用 `Glob` 或 `Read` 检查文件是否存在
3. 如果引用包含标记名（`:marker_name`），检查目标文件中是否存在 `// --8<-- [start:marker_name]` 和 `// --8<-- [end:marker_name]`

**输出格式**：

```
🔴 [代码引用] 第 X 行：文件不存在 — "code/java/javase/io/io-xxx/..."
🔴 [代码引用] 第 X 行：标记不存在 — ":some_marker" 在目标文件中未找到
```

---

## 审查维度 6：格式与元数据合规

**目标**：确保文档排版风格和元数据符合项目规范。格式问题自动修复，元数据问题标记后修复。

### 6A. Front Matter 校验

| 检查项 | 规范 | 方法 |
|--------|------|------|
| `title` 与 nav 一致 | `title:` 值必须与 `zensical.toml` nav 中对应条目的标题完全一致 | 从 nav 中找到文档路径对应的标题，与 `title:` 对比 |
| `icon` 仅二级目录 | 只有 `docs/分类/子分类/index.md` 级别才有 `icon:` 字段 | 检查文档路径层级 |
| 目录结构合规 | 文档路径必须是 `分类/文件夹名/index.md` 形式 | 检查路径最后一段是否为 `index.md` |

### 6B. Nav 注册校验

1. 从文档路径中提取相对路径（去掉 `docs/` 前缀）
2. 在 `zensical.toml` 的 `nav` 数组中搜索该路径
3. 排除 `docs/custom/`、`docs/assert/` 等资源目录

### 6C. 排版风格检查（自动修复）

以下排版问题直接用 `Edit` 修复，无需确认：

| 规范 | 检查方法 |
|------|---------|
| 强调用反引号 `` ` `` | 搜索 `**技术术语**` 或 `*技术术语*` |
| 交叉引用用「」 | 搜索"第X章"或"见第X节" |
| 代码块 ``` 后有空格 | 带属性时检查 ``` 与语言标识符间 |
| Admonition 空行+缩进 | `!!!`/`???` 标题行与内容间有空行，内容缩进 4 格 |
| 列表前有空行 | 段落/文字与无序列表之间必须有一行空行 |
| 图片用 `<figure>` 格式 | 检查是否有裸 `![]()` 图片语法 |

**输出格式**：

```
🔴 [元数据] title 不一致：front matter "插件系统" vs nav "插件"
🔴 [元数据] 路径未注册：docs/topic/oauth/xxx/index.md 未出现在 zensical.toml nav 中
✅ [排版] 已修复：第 X 行，技术术语改为反引号包裹
✅ [排版] 已修复：第 X 行，admonition 内容已补齐 4 格缩进
```

## 审查维度 7：代码增强（自动执行）

**目标**：为手写笔记中的简短代码/命令添加详细注释和具体示例。

用户手写笔记时可能只记录了命令或语法的核心形式（如 `git add xxx`），缺少注释和完整示例。此维度**自动增强**这些内容。

### 7A. 代码注释增强

扫描文档中所有代码块，对缺少注释的代码执行以下操作：

1. **识别无注释代码块**：代码行数 ≥3 但没有任何注释（`//`、`#`、`/* */`）的代码块
2. **为关键行添加中文注释**：解释每一行的作用，特别是参数含义和返回值
3. **校验语法规范性**：检查代码语法是否正确、参数是否完整

**增强示例**：

原文（用户手写）：

```
git add xxx
```

增强后：

```
# 将指定文件添加到暂存区（staging area）
# xxx 可以是具体文件名、目录名或通配符
git add xxx
```

### 7B. 示例补充

对只记录了语法定义但没有具体示例的代码块，在代码块后追加一个带具体参数的完整示例：

1. **识别纯语法块**：包含 `xxx`、`<placeholder>`、`...` 等占位符的代码块
2. **追加具体示例**：用真实值替换占位符，展示实际使用场景
3. **校验示例正确性**：通过 Context7 或 WebSearch 确认示例中的 API/命令用法正确

**增强示例**：

原文：

```
git add xxx
```

追加示例：

```
# 添加单个文件
git add README.md

# 添加整个目录
git add src/

# 添加所有修改的文件
git add .

# 交互式选择要添加的内容
git add -p
```

### 7C. 命令/语法定义校验

对于文档中记录的命令行、API 调用、配置语法等：

1. **校验语法格式**：通过 Context7/WebSearch 确认语法是否规范（参数名、顺序、可选/必选）
2. **补充缺失参数说明**：如果只记录了核心参数，补充常用可选参数
3. **标注版本差异**：如果某语法在不同版本间有变化，添加版本说明

**输出格式**：

```
✅ [代码增强] 第 X 行：已为 git add 命令添加注释和 4 个具体示例
✅ [代码增强] 第 X 行：已为代码块添加关键行注释
🟡 [代码增强] 第 X 行：语法 `xxx` 的参数顺序与 man page 不一致，已修正
```

## 第四步：生成审查报告

按以下格式汇总所有发现和改动：

```markdown

| # | 维度 | 行号 | 问题 | 修复建议 |
|---|------|------|------|---------|

## 🟠 待确认（X 处）— TOC 结构调整

| # | 维度 | 位置 | 当前 | 建议改为 | 原因 |
|---|------|------|------|---------|------|
| 1 | TOC | L50 H2 | `## 概述` | `## Docker 概述` | 孤立术语 |

### 格式排版（X 处）

| # | 行号 | 修复内容 |

### 代码增强（X 处）

| # | 行号 | 增强内容 |
|---|------|---------|

## ✅ 审查通过

| 维度 | 状态 | 详情 |
|------|------|------|

```

## 第五步：执行修复

根据维度分类执行不同策略：

### 自动执行（无需确认）

1. **排版修复**：维度 6C 中的格式问题直接用 `Edit` 修复
2. **代码增强**：维度 7 的注释补充和示例添加直接用 `Edit` 写入
3. **技术事实修正**：维度 1 中查证确认的错误直接修正

### 征求确认后执行

1. **TOC 结构调整**：维度 3 的标题层级变更，用 `AskUserQuestion` 逐条展示建议，用户确认后才执行
2. **元数据修复**：维度 6A/6B 的 front matter 和 nav 注册问题，修复前展示具体变更

### 验证

修复完成后执行 `zensical build` 验证站点构建。
